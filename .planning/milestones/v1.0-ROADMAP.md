# Milestone v1.0: Conference Operations Platform

**Status:** SHIPPED 2026-04-04
**Phases:** 1-7
**Total Plans:** 21

## Overview

v1.0 turned the existing brownfield conference app into a complete single-conference operations platform. The milestone closed the main product gaps around legally defensible onboarding, organizer-controlled program data, attendance-aware participant journeys, self-service recovery, file-capable communication, readiness-aware materials, and consistent conference branding across devices.

## Phases

### Phase 1: Registration & Consent

**Goal**: Users can legally register, choose attendance format, and maintain the participant profile data needed for the conference.
**Depends on**: Nothing (first phase)
**Plans**: 3 plans

Plans:

- [x] 01-01: Backend registration contract, validation, and regression tests
- [x] 01-02: Consent-aware registration UI and auth feedback
- [x] 01-03: Dashboard profile contract alignment and cached user refresh

**Details:**
Registration now captures attendance format, section choice, talk title, and explicit dual consent with versioned audit logging. Profile editing reuses the same contract, and backend validation no longer depends on room assignment existing before schedule planning.

### Phase 2: Authoritative Program Management

**Goal**: Organizers can correct participant-submitted data and assemble an authoritative conference program.
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:

- [x] 02-01: Authoritative program backend and assignment model
- [x] 02-02: Admin program approval UI
- [x] 02-03: Official schedule readers and program PDFs

**Details:**
Organizers can now approve or edit attendance format, section, talk title, room, time slot, and external join link in one admin workflow. Official schedule readers and generated program documents use approved assignments instead of raw participant profile data.

### Phase 3: Hybrid Schedule Experience

**Goal**: Participants can navigate the official program in a way that matches whether they attend online or offline.
**Depends on**: Phase 2
**Plans**: 3 plans

Plans:

- [x] 03-01: Authoritative participant schedule API
- [x] 03-02: Hybrid dashboard schedule UX
- [x] 03-03: Venue navigation aligned to attendance mode

**Details:**
Participants now see authoritative approved placements with explicit pending state. Offline users receive room and venue navigation, while online users receive join links without being pushed through physical-location flows.

### Phase 4: Self-Service Account Recovery

**Goal**: Users can recover account access securely without organizer-led manual resets.
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:

- [x] 04-01: Forgot-password request flow
- [x] 04-02: Reset-password completion API
- [x] 04-03: Public recovery pages and route wiring

**Details:**
Recovery is now end-to-end with opaque reset tokens, sender abstraction support, generic outward responses, single-use reset completion, and public request/reset pages connected from the login flow.

### Phase 5: Conference Communication

**Goal**: Participants can communicate inside the platform with text, safe file sharing, and structured feedback.
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:

- [x] 05-01: Chat attachment backend
- [x] 05-02: Chat attachment UX
- [x] 05-03: Feedback flow completion

**Details:**
Chat now supports allowlisted file attachments, multipart sends, authenticated downloads, and scope-based access checks. Feedback became an end-to-end participant flow with organizer visibility in the admin surface.

### Phase 6: Participant Materials

**Goal**: Participants can retrieve personalized conference materials that reflect final program and participation outcomes.
**Depends on**: Phase 3
**Plans**: 3 plans

Plans:

- [x] 06-01: Document readiness status contract
- [x] 06-02: Policy-aligned document handlers
- [x] 06-03: Status-aware participant materials UI

**Details:**
The platform now exposes readiness before any download attempt, enforces the same attendance-aware policy in each concrete document endpoint, and presents participants with a real material center instead of blind download buttons.

### Phase 7: Branded Responsive Experience

**Goal**: The platform presents a coherent conference identity and works well across devices on public and authenticated pages.
**Depends on**: Phases 1, 3, 5, and 6
**Plans**: 3 plans

Plans:

- [x] 07-01: Shared conference branding and landing content
- [x] 07-02: Responsive participant flow cleanup
- [x] 07-03: Dense authenticated surface cleanup

**Details:**
Shared conference identity now comes from one frontend source backed by `/api/conference`. Public landing content, participant flows, chat, map, and admin surfaces all received branded inline feedback and responsive cleanup.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Store consent as dual versioned audit records so operational and publication consent remain separately provable.
- Treat admin-approved program assignments as the official source of truth for schedule and document generation.
- Branch participant experience by attendance mode at the schedule and navigation level, not just by labels.
- Keep recovery and attachment flows behind explicit security controls: opaque reset tokens, allowlisted files, and authenticated downloads.
- Use `/api/conference` as the shared branding source for public and authenticated frontend surfaces.
- Prefer inline status feedback over browser alerts on primary participant and organizer flows.

**Issues Resolved:**

- Missing legal consent capture and explicit consent logging in registration.
- Official schedule and generated program output depended on raw participant profile data.
- Users had no self-service account recovery.
- Chat had no safe attachment support and feedback lacked organizer visibility.
- Participant documents had no readiness visibility before download.
- Public and authenticated pages lacked a unified conference identity and consistent responsive behavior.

**Issues Deferred:**

- No standalone `v1.0` milestone audit document was produced before archival.
- Automated browser and end-to-end coverage are still missing on the main participant and admin flows.
- Several large page components remain candidates for deeper decomposition.

**Technical Debt Incurred:**

- Chat attachments still rely on local filesystem storage under the application host.
- Public landing content remains code-configured rather than organizer-editable.
- Notifications, audit exports, and attachment moderation were deferred to the next milestone.
